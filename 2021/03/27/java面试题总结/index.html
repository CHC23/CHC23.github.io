java面试题总结

发表于 2021-03-27

1.jdk,jre,jvm区别和联系

jdk:java开发工具。包含jre和java工具（javac,java等）。

jre:java运行环境。包含bin(即jvm)和lib(java类库)

jvm：java虚拟机，把javac编译出的class文件根据java类库解释成机器码在不同平台运行，即跨平台的特性。

2.==和equals()的区别

==：比较的是堆栈中的值，基本类型比较的是栈中的变量值，引用类型比较的是堆中对象的地址。

equals：object中默认也是使用==实现的，不重写的话等价于==，一般都会重写，比如String中equals会重写（重写后比较的是对象的内容，即对象的值）。

3.final

1.final作用：
  • 修饰类：此类不可被继承。
  • 修饰方法：表示方法不可以被子类覆盖，但是可以被重载。
  • 修饰变量：表示变量一但被赋值就不可以重新改变它的值。

（1）修饰成员变量：

  • 修饰类变量：只能在静态初始化块中指定初始值或者在声明该类变量时指定初始值。
  • 修饰成员变量：可以在非静态初始化块中声明该变量或者在构造器中执行初始值。

（2）修饰局部变量：系统不会为局部变量进行初始化，局部变量必须由程序员显式初始化。因此在使用final修饰局部变量时，既可以在定义时指定默认值（后面不能再对其赋值），也可以不指定默认值而是在后面的使用中赋初值（仅一次）。

（3）修饰基本类型和引用类型：

  • 修饰基本类型：数值一旦初始化就不可以再次改变。
  • 引用类型：初始化后就不能再让其指向另一个对象。但是引用的值可以改变
2.为什么局部内部类和匿名内部类只能访问局部final变量

有内部类的文件编译之后会产生两个class文件，内部类和外部类处于同一级别，内部类不会因为定义在方法里面就会随方法的执行完而被销毁。

因此会产生一个问题：

当外部类的方法结束时，局部变量就会被销毁，但是内部类对象可能还存在（只有没有引用时才会死亡），这样就出现了内部类对象访问了一个已经死亡既不存在的变量这种情况。解决的方法是将局部变量复制了一份作为内部类的成员成员变量，当局部变量死亡之后，内部类仍然可以访问它，但是实际访问的该局部变量的“copy”值。

复制局部变量为内部类的成员变量时，必须保证这两个变量的值相同，即当我们改变内部类的成员变量时，方法中的局部变量也得跟着改变，解决的方法就是使用fianl修饰该局部变量，对其初始化之后就不允许修改其值，这样就保证了内部类的成员变量和方法的局部变量的一致性。事实上是一种妥协，使局部变量和内部类内建立的拷贝保持一致性。

4.String、StringBuffer、StringBuilder的区别和使用场景

String是用final修饰的，不可变，每次操作都会产生新的String对象。

StringBuffer和StringBuilder都是在原对象上进行操作。

StringBuffer的方法都是synchrinized修饰的，线程安全。StringBuilder线程不安全，在使用时需要额外使用synchrinized修饰保证线程安全。

性能：StringBuilder>StringBuffer>String。

使用场景：经常需要改变字符串内容时使用StringBuilder和StringBuffer，优先使用StringBuilder，当多线程使用共享变量时使用StringBuffer。

5.重载和重写的区别

重载：发生在同一个类中，是一个类的多态性，方法名必须相同，必须改变参数列表，参数类型不同、个数不同、顺序不同，返回值和访问修饰符可以不同，发生在编译时期

重写：发生在父子类之间，是父子类之间的多态性，方法名和参数列表必须相同，返回值范围小于等于父类，抛出异常的范围小于等于父类，访问修饰符大于等于父类；用private修饰父类的方法时，子类不可以重写该方法。（外壳不变，核心重写）

6.接口和抽象类

相同点

  • 都不可以被实例化。
  • 接口是对动作的抽象，抽象类是对根源的抽象（即事物的本质）。
  • 接口只有被其实现类实现了全部方法才可以被实例化，抽象类只有被其单继承的子类实现了其全部方法才能被实例化。

不同点：

  • 接口只能定义抽象方法（public abstract），不能实现方法。抽象类即可以定义抽象方法，也可以实现方法，也可以有构造方法。
  • 接口多实现，抽象类单继承。
  • 接口的成员变量只能是public static final类型的，抽象类的成员变量可以是各种类型。

7.List和Set的区别

List：有序，按照对象进入的顺序保存对象，可以重复对象，可以存放对个NULL对象。使用get（int index）获取指定下标的元素，还可以使用lterator取出所有元素载逐一遍历。

Set：无序存放，不能重复，最多允许有一个Null对象，只能使用lterator取出所有元素再逐一遍历。

8.hashCode和equals

hashCode()

hashCode()根据哈希函数计算出某对象在堆中的位置（在堆里面维护了一张哈希表）,返回对应的哈希码（一个整数）。hashCode（）定义在object类中，java的任何类都包含hashCode（）。用来比较两个对象是否一致。

hashCode和equals

重写的equals（）在比较使用时更加复杂全面，所以效率低。而由于不同的对象计算出来的哈希码也可能相同，即产生了哈希冲突。由此有这样两个问题：

  • 使用equals()相等的两个对象，他们的哈希码一定相同，equals()绝对可靠，但是效率相对低。
  • 使用hashCode()会产生哈希冲突，相对不可靠。

解决办法是，当需要比较时，先试用hashCode()进行比较，如果返回的哈希码不相同，那么比较的对象肯定不相等，此时不需要再去使用equals()进行对比。当返回的哈希码相同时，再使用equals()进行对比，如果equals()相同，则比较的对象相等，反之不相等。这样使用既提高了可靠性也保证了效率。

应该遵守的规则
  • 两个对象相等，分别调用equals()方法时都会返回true。并且二者的hashCode也一定相同。
  • 两个对象有相同的hashCode，但它们不一定相同
  • equals()重写之后比较的是对象的内容，所以当仅仅重写equals()而未重写hashCode()时，会出现equals相等的对象，hashCode不相等的情况。为了避免这种情况，在重写equals()的同时也要重写hashCode()。
  • hashCode的默认行为是对堆上的对象产生独特值，如果没有重写hashCode()，则该class的两个对象无论如何都不会相等（即使这两个对象指向相同的数据）。

9.ArrayList和LinkedList区别

ArrayList：底层实现是一个动态数组，需要连续内存存储，支持下标访问（随机访问）。其扩容机制：数组长度固定，当超出数组存储长度时需要进行扩容，通过调用Arrays.copyOf()把原数组拷贝到建立的新数组，然后插入新的数据（使用尾插法不会涉及元素的移动）。新数组的容量是原数组的1.5倍（原数组长度为偶数则是1.5倍，为奇数则是1.5倍-0.5）。由于需要拷贝整个数组，所以代价很高，最好在创建ArrayList对象时就指定大概的容量大小，减少扩容的次数。删除操作需要移动元素，时间复杂度为O(n)，代价非常高。

LinkedList：基于双向链表实现，不需要连续存储空间。其查询需要遍历链表，时间复杂度为O(n)，因此不适合查询。插入和删除操作不涉及元素的移动，因此适合插入和删除操作。遍历操作必须使用iterator，不能使用for循环，因为for循环遍历时，每次通过get(i)取得某一元素时都要对list重新遍历，性能消耗大。也不要使用indexOf返回元素索引进行遍历，因为会对整个list进行遍历，即使结果为空时也会遍历整个list.

10.HashMap和HashTable的底层实现，区别

HashMap使用拉链法解决哈希冲突，所以它的底层是一个“数组加链表”，无冲突的放入数组，发生冲突了则放在对应的链表。jdk8以后当链表长度超过8就会转化为红黑树（自平衡的二叉查找树），小于6时转回为链表。

二者区别：

  • HashMap没有使用synchrinized修饰，线程不安全。后者线程安全。
  • HashMap允许键值为空。后者不允许。

11.ConcurrentHashMap

jdk7中

数据结构：ReentrantLock+Segment+HashEntry，一个Segment中包含一个HashEntry数组，拉链法解决哈希冲突。

元素查询：二次Hash，第一次定位到Segment，第二次定位到元素所在的头部。

锁：Segment分段锁，Segment继承ReentrantLock，锁定操作的Segment，其他的Segment不受影响。并发度为Segment的个数，可以通过构造函数指定，数组扩容不会影响其他的Segment。get方法无需加锁，volatile保证。

jdk8中

数据结构：synchronized+CAS(一种乐观锁机制)+Node+红黑树，Node的val和next都用volatile修饰，办证可见性。查找、替换、赋值都用CAS。

锁：锁链表的head节点，不影响其他元素的读写，所粒度更细、效率更高（相比jdk7只锁了一个链表）。扩容时，阻塞所有的读写操作、并发扩容。

读写操作无锁，Node的val和next使用volatile修饰，读写线程对该变量相互可见，数组使用volatile修饰，保证扩容时被读线程指道。

12.java字节码

java虚拟机（jvm）是机器和编译程序之间的一层虚拟的机器，jvm在任何平台上都提供给编译程序一个共同的接口。

java程序由javac编译成字节码，供虚拟机理解（只面向虚拟机），再有虚拟机解释成特定平台的机器码。每一种平台的解释器不同，但现实的虚拟机是相同的。java中编译和解释并存。

java代码→编译器（javac）→字节码→jvm→解释器→二进制机器码→程序运行。

字节码优点：

  • 在一定程度上解决了解释型语言执行效率低的问题，同时保留了解释型语言可移植的特点。
  • 字节码只面向虚拟机，不特指系统平台，因此java程序无需重新编译就可以在不同计算机系统上运行。

13.java类加器

类加载器作用：将java的类动态的加载到jvm中执行

jdk自带三个类加载器：

  • bootstrapClassLoader（根类加载器）：最顶层的类加载器，默认负责加载%JAVA_HOME%lib下的jar包和class文件。
  • ExtClassLoader（扩展类加载器）：继承bootstrapClassLoader，负责加载%JAVA_HOME%lib/ext下jar包和class文件。
  • AppClassLoader（系统类加载器）:继承ExtClassLoader，负责加载Classpath下的类文件。是系统类加载器，线程上下文加载器。
  • 继承ClassLoader实现自定义类加载器。
